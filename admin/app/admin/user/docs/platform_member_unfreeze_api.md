# 平台会员解冻 API 文档

## 接口说明

实现管理后台会员解冻功能，管理员可对已冻结用户进行解冻操作，解冻后会员权益立即恢复。

## 接口地址

```
POST /api/v1/platform-member/unfreeze
```

## 请求参数

### Header

```
Authorization: Bearer <admin_access_token>
Content-Type: application/json
```

### 请求 Body

```json
{
  "userId": 1001,
  "unfreezeReason": "用户申诉成功，经核查违规事实不成立，予以解冻恢复会员权益"
}
```

### 参数说明

| 参数名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| userId | int64 | 是 | 用户 ID | 1001 |
| unfreezeReason | string | 是 | 解冻原因（必填，用于审计） | 用户申诉成功 |

### 请求示例

```bash
curl -X POST 'http://localhost:8000/api/v1/platform-member/unfreeze' \
  -H 'Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...' \
  -H 'Content-Type: application/json' \
  -d '{
    "userId": 1001,
    "unfreezeReason": "用户申诉成功，经核查违规事实不成立，予以解冻恢复会员权益"
  }'
```

## 成功响应

### 响应数据结构

```json
{
  "code": 200,
  "msg": "解冻成功",
  "data": {
    "userId": 1001,
    "memberLevel": 2,
    "memberLevelName": "SVIP 会员",
    "memberStatus": 0,
    "expireTime": 1735689600,
    "unfreezeTime": 1704153600,

    "operatorId": 100,
    "updateTime": 1704153600

```

### 字段说明

| 字段名 | 类型 | 说明 |
|--------|------|------|
| userId | int64 | 用户 ID |
| memberLevel | int32 | 会员等级（解冻前等级，保持不变） |
| memberLevelName | string | 会员等级名称 |
| memberStatus | int32 | 会员状态（0-正常） |
| expireTime | int64 | 会员过期时间戳（保持不变） |
| unfreezeTime | int64 | 解冻时间戳 |
| unfreezeReason | string | 解冻原因 |
| operatorId | int64 | 操作管理员 ID |
| updateTime | int64 | 更新时间戳 |

## 错误响应

```json

  "msg": "参数异常",
  "data": null
}
```

常见错误：
- 解冻原因不能为空
- 用户账号状态异常
- 用户未开通会员
- 会员未冻结，无需解冻

  "msg": "未授权",

## 业务逻辑

### 1. 操作流程

```
管理员请求 → 校验权限 → 校验用户状态 → 校验会员冻结状态 → 
执行解冻 → 恢复状态 → 记录日志 → 刷新权益 → 返回结果
```

### 2. 参数校验

**必填参数校验**
- `userId`：非空，用户必须存在
- `unfreezeReason`：解冻原因必填（审计要求）

**会员状态校验**
- ✅ 仅可解冻已冻结的会员
- ❌ 禁止解冻未开通会员的用户
- ❌ 禁止解冻已过期的会员
- ❌ 禁止解冻正常状态的会员

### 3. 解冻核心逻辑

**状态恢复**
- 会员状态更新为 `0`（正常）
- 保留原会员等级（不变）
- 保留原到期时间（不变）
- 记录解冻时间

**权益恢复**
- 清除 Redis 旧会员权益缓存
- 重新加载当前有效会员权益
- 设备绑定上限立即恢复
- 高音质等内容权限立即恢复

**操作日志**
- 管理员 ID
- 用户 ID
- 操作类型（解冻）
- 解冻原因
- 操作时间

### 4. 数据更新

**更新会员表（user_member）**
- 更新 `status` = 0（正常状态）
- 更新 `updated_at`（最后操作时间）

**记录操作日志（member_operate_log）**
- 创建解冻操作记录
- 记录解冻原因

## 会员状态说明

| 状态码 | 状态 | 说明 | 可执行操作 |
|--------|------|------|-----------|
| 0 | 正常 | 会员正常使用中 | 可冻结 |
| 1 | 已过期 | 会员已过期 | 可续费 |
| 2 | 已冻结 | 会员被管理员冻结 | 可解冻 |

## 异常处理

| 异常类型 | 处理方式 |
|----------|----------|
| 未登录 | 返回 401 未授权 |
| 无权限 | 返回 403 权限不足 |
| 用户不存在 | 返回 404 用户不存在 |
| 解冻原因缺失 | 返回 400 参数异常 |
| 用户未开通会员 | 返回 400 不可解冻 |
| 会员未冻结 | 返回 400 无需解冻 |
| 用户账号异常 | 返回 400 参数异常 |

## 约束规则

### 1. 权限约束

- ✅ 只允许管理员操作
- ✅ 需要「会员管理 - 解冻」权限
- ✅ 普通用户禁止访问

### 2. 审计约束

- ✅ 解冻原因必填（用于追溯）
- ✅ 每次解冻都留痕
- ✅ 支持后台查看操作日志
- ✅ 记录操作管理员 ID

### 3. 状态约束

- ✅ 仅可解冻已冻结状态的会员
- ✅ 解冻后会员立即恢复可用
- ✅ 不修改等级和时长
- ✅ 禁止重复解冻

### 4. 实时性约束

- ✅ 解冻立即生效
- ✅ 权益实时恢复
- ✅ 权限实时更新

## 操作日志查询

管理员可以在后台查看会员解冻历史：

```sql
SELECT 
  operate_time,
  operate_admin,
  user_id,
  operate_type,
  old_level,
  new_level,
  old_expire_time,
  new_expire_time,
  remark
FROM member_operate_log
WHERE user_id = 1001
  AND operate_type = 3  -- 解冻操作
ORDER BY operate_time DESC
LIMIT 20;
```

## 最佳实践

### 1. 解冻原因规范

建议解冻原因包含以下信息：
- 解冻依据（申诉成功、核实无误、用户申请等）
- 核查情况（已核查、证据不足、误判等）
- 处理决定（予以解冻、恢复权益等）

**示例**
- ✅ "用户申诉成功，经核查违规事实不成立，予以解冻恢复会员权益"
- ✅ "用户主动申请解冻，因长期不使用账号，现申请恢复使用"
- ✅ "系统误判，经技术核查为系统 bug 导致，予以解冻并补偿 30 天"
- ❌ "解冻"（过于简单）
- ❌ ""（空原因）

### 2. 解冻场景

**适用场景**
- 用户申诉成功，违规事实不成立
- 系统误判，技术问题导致错误冻结
- 用户主动申请解冻（如长期不用后申请恢复）
- 临时冻结后核查无误
- 其他合理解冻理由

**不适用场景**
- 会员未冻结（无需解冻）
- 会员已过期（应续费而非解冻）
- 用户未开通会员（无需解冻）

### 3. 解冻时机

- 申诉处理完成后立即解冻
- 核查无误后及时解冻
- 用户申请审核后酌情解冻
- 系统故障修复后批量解冻

### 4. 批量操作

对于批量解冻：
- 需要更高级别权限审批
- 建议逐个操作并详细记录原因
- 操作后抽查验证
- 定期审计解冻记录

## 冻结/解冻关联

### 1. 状态流转

```
正常 (0) ←→ 已冻结 (2)
  ↓
已过期 (1)
```

### 2. 操作对应

| 操作 | 接口 | 状态变化 | 权益变化 |
|------|------|---------|---------|
| 冻结 | POST /freeze | 正常 → 冻结 | 立即失效 |
| 解冻 | POST /unfreeze | 冻结 → 正常 | 立即恢复 |

### 3. 权益对比

| 权益类型 | 冻结状态 | 解冻状态 |
|---------|---------|---------|
| 设备绑定上限 | ❌ 失效 | ✅ 恢复 |
| 高音质权限 | ❌ 失效 | ✅ 恢复 |
| 内容权限 | ❌ 失效 | ✅ 恢复 |
| 会员价格 | ❌ 失效 | ✅ 恢复 |
| 专属客服 | ❌ 失效 | ✅ 恢复 |

### 2. 操作日志页

在操作日志页展示：
- 所有冻结/解冻记录
- 操作管理员信息
- 冻结/解冻原因详情

### 3. 数据统计

在管理后台统计：
- 当前冻结会员数量
- 今日冻结/解冻数量
- 冻结/解冻原因分布
- 高频冻结/解冻管理员
- 解冻成功率统计

在申诉管理页：
- 用户申诉列表
- 申诉审核状态
- 审核通过后自动解冻
- 审核拒绝后维持冻结

## 审计要求

### 1. 操作留痕

每次解冻操作必须记录：
- 操作时间
- 操作管理员
- 解冻原因
- 用户 ID
- 操作前后状态

### 2. 原因追溯

解冻原因必须详细记录：
- 申诉成功的，记录申诉编号
- 系统误判的，记录故障编号
- 用户申请的，记录申请理由
- 其他原因的，详细说明

### 3. 定期审计

建议定期审计：
- 每周审查解冻记录
- 每月统计解冻原因分布
- 每季度分析冻结/解冻比例
- 每年评估冻结策略合理性
