# 平台会员冻结 API 文档

## 接口说明

实现管理后台会员冻结功能，管理员可对违规用户进行会员冻结操作，冻结后会员权益立即失效。

## 接口地址

```
POST /api/v1/platform-member/freeze
```

## 请求参数

### Header

```
Authorization: Bearer <admin_access_token>
Content-Type: application/json
```

### 请求 Body

```json
{
  "userId": 1001,
  "freezeReason": "用户违规刷取会员权益，经核查属实，予以冻结处理"
}
```

### 参数说明

| 参数名 | 类型 | 必填 | 说明 | 示例 |
|--------|------|------|------|------|
| userId | int64 | 是 | 用户 ID | 1001 |
| freezeReason | string | 是 | 冻结原因（必填，用于审计） | 用户违规刷取会员权益 |

### 请求示例

```bash
curl -X POST 'http://localhost:8000/api/v1/platform-member/freeze' \
  -H 'Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9...' \
  -H 'Content-Type: application/json' \
  -d '{
    "userId": 1001,
    "freezeReason": "用户违规刷取会员权益，经核查属实，予以冻结处理"
  }'
```

## 成功响应

### 响应数据结构

```json
{
  "code": 200,
  "msg": "冻结成功",
  "data": {
    "userId": 1001,
    "memberLevel": 2,
    "memberLevelName": "SVIP 会员",
    "memberStatus": 2,
    "expireTime": 1735689600,
    "freezeTime": 1704067200,
    "freezeReason": "用户违规刷取会员权益，经核查属实，予以冻结处理",
    "operatorId": 100,
    "updateTime": 1704067200
  }
}
```

### 字段说明

| 字段名 | 类型 | 说明 |
|--------|------|------|
| userId | int64 | 用户 ID |
| memberLevel | int32 | 会员等级（冻结前等级） |
| memberLevelName | string | 会员等级名称 |
| memberStatus | int32 | 会员状态（2-冻结） |
| expireTime | int64 | 会员过期时间戳（保留原到期时间） |
| freezeTime | int64 | 冻结时间戳 |
| freezeReason | string | 冻结原因 |
| operatorId | int64 | 操作管理员 ID |
| updateTime | int64 | 更新时间戳 |

## 业务逻辑

### 1. 操作流程

```
管理员请求 → 校验权限 → 校验用户状态 → 校验会员状态 → 
执行冻结 → 更新状态 → 记录日志 → 清除权益 → 返回结果
```

### 2. 参数校验

**必填参数校验**
- `userId`：非空，用户必须存在
- `freezeReason`：冻结原因必填（审计要求）

**会员状态校验**
- ✅ 仅可冻结正常/未过期的会员
- ❌ 禁止冻结未开通会员的用户
- ❌ 禁止冻结已过期的会员
- ❌ 禁止冻结已冻结的会员

### 3. 冻结核心逻辑

**状态更新**
- 会员状态更新为 `2`（冻结）
- 保留原会员等级
- 保留原到期时间（解冻可恢复）
- 记录冻结时间

**权益失效**
- 立即清除 Redis 会员权益缓存
- 权益校验接口返回无权限
- 设备绑定上限立即失效
- 高音质等内容权限立即失效

**操作日志**
- 管理员 ID
- 用户 ID
- 操作类型（冻结）
- 原等级/新等级（不变）
- 原到期时间/新到期时间（不变）
- 冻结原因
- 操作时间

### 4. 数据更新

**更新会员表（user_member）**
- 更新 `status` = 2（冻结状态）
- 更新 `updated_at`（最后操作时间）

**记录操作日志（member_operate_log）**
- 创建冻结操作记录
- 记录冻结原因

## 会员状态说明

| 状态码 | 状态 | 说明 |
|--------|------|------|
| 0 | 正常 | 会员正常使用中 |
| 1 | 已过期 | 会员已过期 |
| 2 | 已冻结 | 会员被管理员冻结 |

## 异常处理

| 异常类型 | 处理方式 |
|----------|----------|
| 未登录 | 返回 401 未授权 |
| 无权限 | 返回 403 权限不足 |
| 用户不存在 | 返回 404 用户不存在 |
| 冻结原因缺失 | 返回 400 参数异常 |
| 用户未开通会员 | 返回 400 不可冻结 |
| 会员已过期 | 返回 400 无需冻结 |
| 会员已冻结 | 返回 400 重复冻结 |
| 用户账号异常 | 返回 400 参数异常 |

## 约束规则

### 1. 权限约束

- ✅ 只允许管理员操作
- ✅ 需要「会员管理 - 冻结」权限
- ✅ 普通用户禁止访问

### 2. 审计约束

- ✅ 冻结原因必填（用于追溯）
- ✅ 每次冻结都留痕
- ✅ 支持后台查看操作日志
- ✅ 记录操作管理员 ID

### 3. 状态约束

- ✅ 冻结后会员立即不可用
- ✅ 保留原等级和到期时间
- ✅ 解冻后可恢复原时长
- ✅ 禁止重复冻结

### 4. 实时性约束

- ✅ 冻结立即生效
- ✅ 权益立即失效
- ✅ 设备上限实时更新

## 数据表说明

### 主要数据表

- `users`：用户基础信息表
- `user_member`：会员信息表
- `member_operate_log`：会员操作日志表

### 字段映射

| 请求字段 | 数据表 | 字段名 |
|----------|--------|--------|
| userId | users | user_id |
| freezeReason | member_operate_log | remark |
| operatorId | member_operate_log | operate_admin |

| freezeTime | member_operate_log | operate_time |

## 操作日志查询

管理员可以在后台查看会员冻结历史：

```sql
SELECT 
  operate_time,
  operate_admin,
  user_id,
  operate_type,
  old_level,
  new_level,
  old_expire_time,
  new_expire_time,
  remark
FROM member_operate_log
WHERE user_id = 1001
  AND operate_type = 2  -- 冻结操作
ORDER BY operate_time DESC
LIMIT 20;
```

## 最佳实践

### 1. 冻结原因规范

建议冻结原因包含以下信息：
- 违规类型（刷权益、恶意退款、账号异常等）
- 核查情况（已核查、证据确凿等）
- 处理依据（平台规则第 X 条）

**示例**
- ✅ "用户违规刷取会员权益，经核查属实，予以冻结处理"
- ✅ "用户恶意退款，涉及金额 XXX 元，根据平台规则予以冻结"
- ✅ "账号存在异常登录行为，为保护用户权益临时冻结"
- ❌ "违规"（过于简单）
- ❌ ""（空原因）

### 2. 冻结场景

**适用场景**
- 用户违规刷取会员权益
- 恶意退款/拒付
- 账号被盗/异常登录
- 用户主动申请冻结（如长期不用）
- 其他违反平台规则的行为

**不适用场景**
- 会员已过期（无需冻结）
- 会员已被冻结（禁止重复）
- 用户未开通会员（无需冻结）

### 3. 冻结时长

- 临时冻结：用于账号安全问题，待用户申诉后解冻
- 永久冻结：用于严重违规行为
- 建议根据违规程度选择冻结时长

### 4. 批量操作

对于批量冻结：
- 需要更高级别权限审批
- 建议逐个操作并详细记录原因
- 操作后抽查验证
- 定期审计冻结记录

## 解冻说明

冻结后可通过解冻接口恢复会员：

**解冻接口**
```
POST /api/v1/platform-member/unfreeze
```

**解冻逻辑**
- 恢复会员状态为正常（0）
- 保留原等级和到期时间
- 恢复会员权益
- 记录解冻操作日志
