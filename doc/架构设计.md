# 项目架构文档

## 1. 系统架构

### 1.1 整体架构

```
┌─────────────┐
│   Client    │
└──────┬──────┘
       │
┌──────▼──────────────────────────────────┐
│         API Gateway (AWS ALB)           │
└──────┬──────────────────────────────────┘
       │
┌──────▼──────┬──────────┬──────────┬─────┐
│ User Service│Device Svc│Content Svc│Admin│
└──────┬──────┴──────┬───┴──────┬───┴─────┘
       │             │          │
┌──────▼─────────────▼──────────▼─────────┐
│         PostgreSQL (Aurora)              │
└──────────────────────────────────────────┘
       │
┌──────▼─────────────────────────────────┐
│         Redis (ElastiCache)            │
└────────────────────────────────────────┘
       │
┌──────▼─────────────────────────────────┐
│         Kafka / SQS                    │
└────────────────────────────────────────┘
```

### 1.2 微服务划分

#### 用户服务 (User Service)
- 端口: 8001
- 功能: 用户注册、登录、认证、权限管理
- 数据库: users, user_auth, roles, user_role_rel

#### 设备服务 (Device Service)
- 端口: 8002
- 功能: 设备管理、绑定、状态上报、指令下发
- 数据库: devices, device_user_rel, device_commands
- 缓存: Redis (设备实时状态)

#### 内容服务 (Content Service)
- 端口: 8003
- 功能: 音频上传、内容管理、播放列表
- 数据库: raw_contents, processed_contents, contents, content_play_records
- 存储: S3

#### 媒体处理服务 (Media Processing Service)
- 端口: 8004
- 功能: 接收上传任务，发送到 SQS
- 消息队列: SQS

#### AI Worker
- 功能: 从 SQS 拉取任务，执行 AI 推理，上传处理后的音频
- 运行环境: EC2 / ECS

#### 后台管理 (Admin)
- 端口: 8000
- 功能: 系统管理、用户管理、内容审核、数据统计

## 2. 数据流

### 2.1 音频上传流程

```
1. 用户请求上传 → Content Service
2. Content Service 生成预签名 URL → 返回给用户
3. 用户直接上传到 S3
4. 上传完成后通知 Content Service
5. Content Service 创建 raw_content 记录
6. 发送 AI 处理任务到 SQS
7. AI Worker 拉取任务
8. AI Worker 下载原始音频
9. AI Worker 执行推理
10. AI Worker 上传处理后的音频到 S3
11. AI Worker 更新 processed_content 记录
12. Content Service 创建 content 记录
```

### 2.2 设备播放流程

```
1. 设备请求播放列表 → Device Service
2. Device Service 查询用户的内容 → Content Service
3. Content Service 返回 CDN URL
4. 设备直接从 CDN 下载音频
5. 设备播放音频
6. 设备上报播放记录 → Device Service
7. Device Service 记录到数据库
```

### 2.3 设备心跳流程

```
1. 设备每 30 秒上报心跳 → Device Service
2. Device Service 更新 Redis 中的设备状态
3. Redis 设置 TTL 60 秒
4. 如果设备超过 60 秒未上报，自动标记为离线
```

## 3. 数据库设计

### 3.1 核心表

#### users (用户表)
- id: 主键
- email: 邮箱（唯一）
- mobile: 手机号
- password: 密码（加密）
- nickname: 昵称
- avatar: 头像
- status: 状态（1=正常，0=禁用）
- created_at, updated_at

#### devices (设备表)
- id: 主键
- sn: 设备序列号（唯一）
- model: 设备型号
- firmware_version: 固件版本
- status: 状态（online/offline）
- created_at, updated_at

#### contents (内容表)
- id: 主键
- processed_content_id: 处理后内容 ID
- title: 标题
- duration: 时长
- status: 状态（online/offline）
- created_at

### 3.2 关系设计

- 用户 ←→ 设备：多对多（device_user_rel）
- 用户 ←→ 角色：多对多（user_role_rel）
- 原始内容 → 处理后内容：一对多
- 处理后内容 → 最终内容：一对一

## 4. Redis 数据结构

### 4.1 设备状态

```
Key: device:status:{device_id}
Type: Hash
Fields:
  - online: 1/0
  - battery: 电量百分比
  - volume: 音量
  - play_status: 播放状态
  - timestamp: 时间戳
TTL: 60 秒
```

### 4.2 离线指令

```
Key: device:commands:{device_id}
Type: List
Value: JSON 格式的指令
TTL: 7 天
```

### 4.3 用户会话

```
Key: user:session:{user_id}
Type: String
Value: JWT Token
TTL: 1 小时
```

## 6. API 设计规范

### 6.1 RESTful 风格

- GET: 查询
- POST: 创建
- PUT: 更新
- DELETE: 删除

### 6.2 URL 规范

```
/api/v1/{service}/{resource}
```

示例：
- POST /api/v1/user/register
- GET /api/v1/device/list
- POST /api/v1/content/upload

### 6.3 响应格式

成功：
```json
{
  "code": 0,
  "msg": "success",
  "data": {}
}
```

失败：
```json
{
  "code": 1001,
  "msg": "error message",
  "data": null
}
```

## 7. 认证和鉴权

### 7.1 JWT 认证

- 用户登录后返回 access_token 和 refresh_token
- access_token 有效期 1 小时
- refresh_token 有效期 7 天
- 请求头: `Authorization: Bearer {token}`

### 7.2 权限控制

- 基于角色的访问控制（RBAC）
- 角色：admin, user
- 权限存储在 roles 表的 permissions 字段（JSONB）

## 8. 监控和日志

### 8.1 日志级别

- DEBUG: 调试信息
- INFO: 一般信息
- WARN: 警告信息
- ERROR: 错误信息
- FATAL: 致命错误

### 8.2 监控指标

- API 响应时间
- 数据库查询时间
- Redis 命中率
- S3 上传/下载速度
- 设备在线率
- AI 处理成功率

### 8.3 告警规则

- API 响应时间 > 1s
- 数据库连接数 > 80%
- Redis 内存使用 > 80%
- 设备离线率 > 20%
- AI 处理失败率 > 10%

## 9. 性能优化

### 9.1 数据库优化

- 索引优化：为常用查询字段添加索引
- 连接池：使用连接池管理数据库连接
- 读写分离：主库写，从库读
- 分页查询：避免一次性查询大量数据

### 9.2 缓存优化

- 热点数据缓存：设备状态、用户会话
- 缓存预热：启动时加载常用数据
- 缓存更新：使用 Redis Pub/Sub 同步缓存

### 9.3 CDN 优化

- 音频文件通过 CDN 分发
- 设置合理的缓存策略
- 使用 HTTP Range 支持断点续传

## 10. 安全措施

### 10.1 数据安全

- 密码加密：使用 bcrypt
- 数据传输：HTTPS
- 敏感数据：加密存储

### 10.2 API 安全

- 限流：防止 DDoS 攻击
- 签名验证：防止篡改
- IP 白名单：限制访问来源

### 10.3 设备安全

- 设备认证：基于 SN 和密钥
- 固件签名：防止固件被篡改
- OTA 更新：支持远程升级

## 11. 扩展性

### 11.1 水平扩展

- 微服务无状态设计
- 使用负载均衡器分发请求
- 数据库读写分离和分库分表

### 11.2 垂直扩展

- 升级服务器配置
- 优化代码性能
- 使用更高性能的数据库

## 12. 灾备方案

### 12.1 数据备份

- 数据库：每日全量备份 + 实时增量备份
- S3：启用版本控制和跨区域复制
- Redis：RDB + AOF 持久化

### 12.2 故障恢复

- 主备切换：自动故障转移
- 数据恢复：从备份恢复
- 服务降级：关闭非核心功能
